## 1. CPU とは何か

**CPU (Central Processing Unit)** はコンピュータの「演算装置」。
プログラムの**命令を実行する主体**であり、メモリから読んだデータを処理して結果をメモリに書き戻す。

### 何が起きてるか（超簡略化）

```
[プログラム（メモリ上）]
   ┌──────────────────────────┐
   │ ADD r1, r2, r3           │ ← この命令を CPU が読む
   │ MOV [addr], r1           │
   │ ...                      │
   └──────────────────────────┘
            │
            ▼
        [CPU]
        ┌─────────────────────────┐
        │  1 秒間に数十億回        │
        │  命令を実行する          │
        └─────────────────────────┘
```

現代の CPU は **数 GHz**（= 1 秒に数十億サイクル）で動作する。  
1 命令が 1 サイクルとは限らない（実際は **1 サイクルに複数命令** or 1 命令に複数サイクル、平均で IPC ≈ 1〜4）。

### CPU と他のリソースの関係

CPU は単独では何もできない。以下と協調する：

| リソース | 役割 |
|---|---|
| メモリ (RAM) | CPU が読み書きするデータ・命令の置き場 |
| ディスク / SSD | 永続化された大容量ストレージ |
| ネットワーク | 他のマシン・サービスとの通信 |
| キャッシュ | CPU 内蔵の超高速メモリ（後述）|
| 周辺機器（GPU 等） | 専門的な演算を委譲 |

→ CPU が早く動いても、メモリやディスクが遅いと **CPU は待たされる**（idle / iowait）。

## 2. CPU のハードウェア構造

### 主な構成要素

```
┌──────────────────────────────────────────────────────────────┐
│                       CPU パッケージ                          │
│                                                                │
│  ┌──────────────────────┐    ┌──────────────────────┐         │
│  │      Core 0          │    │      Core 1          │   ...   │
│  │  ┌────┐  ┌────┐      │    │  ┌────┐  ┌────┐      │         │
│  │  │ALU │  │FPU │      │    │  │ALU │  │FPU │      │         │
│  │  └────┘  └────┘      │    │  └────┘  └────┘      │         │
│  │  Registers           │    │  Registers           │         │
│  │  L1 Cache (per core) │    │  L1 Cache (per core) │         │
│  │  L2 Cache            │    │  L2 Cache            │         │
│  └──────────────────────┘    └──────────────────────┘         │
│                                                                │
│  ┌──────────────────────────────────────────────────┐         │
│  │           L3 Cache (全 core で共有)              │         │
│  └──────────────────────────────────────────────────┘         │
│                                                                │
│  Memory Controller → [DRAM (main memory)]                     │
└──────────────────────────────────────────────────────────────┘
```

### 各要素の役割

| 要素 | 役割 |
|---|---|
| **Core** | 命令を実行する基本単位。1 CPU に複数搭載 |
| **ALU** (Arithmetic Logic Unit) | 整数演算・論理演算 |
| **FPU** (Floating Point Unit) | 浮動小数点演算 |
| **Registers** | CPU 内部の超高速メモリ（数十個程度）。命令の作業領域 |
| **L1 / L2 Cache** | core ごとの高速メモリ |
| **L3 Cache** | 全 core で共有する大容量キャッシュ |
| **Memory Controller** | DRAM とのやり取りを管理 |

### メモリ階層

CPU から見た「遅さ」は段階的：

| 階層 | 容量目安 | アクセス時間目安 | サイクル目安 |
|---|---:|---:|---:|
| Register | 数十 | 0.3 ns | 1 サイクル |
| **L1 Cache** | 32-64 KB | ~1 ns | ~4 |
| **L2 Cache** | 256 KB-1 MB | ~3 ns | ~10 |
| **L3 Cache** | 数-数十 MB | ~10 ns | ~40 |
| **DRAM** | 数-数百 GB | ~100 ns | **~300** |
| SSD | 数百 GB-TB | ~100 μs | ~300,000 |
| HDD | TB | ~10 ms | ~30,000,000 |

→ **DRAM アクセスは L1 の 100 倍遅い**。つまり「キャッシュに乗っているか」が性能を支配する。

### SIMD

**Single Instruction, Multiple Data** = 1 命令で複数データを処理する仕組み。
- 例: AVX, AVX-512, ARM NEON
- 暗号化（AES-NI）、JSON パース、ベクトル検索などで活用
- アプリ層から見ると「同じ処理を一度に複数件こなせる」演算高速化

## 3. 命令実行サイクル

CPU は機械語命令を 1 つずつ取り込んで実行する。基本サイクルは 5 段階：

```
1. Fetch    : メモリから命令を取ってくる
   ↓
2. Decode   : 命令の意味を解釈する
   ↓
3. Execute  : ALU/FPU で演算する
   ↓
4. Memory   : 必要ならメモリにアクセス
   ↓
5. Writeback: 結果をレジスタに書き戻す
```

1 命令を素直にやると 5 サイクルかかる。これでは遅いので、CPU は色々な高速化を仕込んでいる（次章）。

## 4. パイプライン・分岐予測・OoO 実行

### パイプライン

複数の命令を**重ね合わせて並列に進める**。

```
時刻        →
命令 1: [Fetch][Decode][Execute][Memory][Writeback]
命令 2:        [Fetch][Decode][Execute][Memory][Writeback]
命令 3:               [Fetch][Decode][Execute][Memory][Writeback]
```

→ 1 サイクルに 1 命令完了するように見える（IPC ≈ 1）。
現代の CPU はパイプライン段数が 10〜20、命令を取り込む幅も複数（スーパースカラ）で、**1 サイクルに数命令完了**できる。

### 分岐予測

`if` 文や `for` ループは「次にどの命令を実行するか」が動的に変わる：

```
if (condition) {
  // 命令 A
} else {
  // 命令 B
}
// 命令 C
```

パイプラインは「次の命令を先取り」するので、分岐がどちらに行くかを **予測**する必要がある。

- 予測当たり → そのまま続行（速い）
- 予測外れ → **パイプラインフラッシュ**（先取りした命令を捨てる）= 10〜20 サイクル損失

```
予測ヒット率は通常 95% 超。だが残り 5% が積み上がると無視できない。
データに依存した分岐（ランダム）は予測しにくい。
```

### Out-of-Order (OoO) 実行

依存関係のない命令を **元の順番に関係なく実行する**：

```
ソース順:                CPU 内部:
  A = read(x)              ← x 読み込み中
  B = read(y)              ← y 読み込み中（並列）
  C = A + B                ← A,B 揃ってから実行
  D = 100 * 200            ← 待たずに先行実行
```

メモリ読み込みの待ち時間に他の命令を進められる。
→ **メモリ局所性が良いと CPU は本領発揮、悪いと idle になる**。

## 5. マルチコア・マルチプロセッサ

### コアとソケット

- 1 つの CPU パッケージに複数 core が入る（マルチコア）
- マザーボードに複数 CPU を載せる構成もある（マルチソケット / SMP）

```
┌──────────────────┐    ┌──────────────────┐
│   Socket 0       │    │   Socket 1       │
│   ┌──┬──┬──┬──┐ │    │   ┌──┬──┬──┬──┐ │
│   │C0│C1│C2│C3│ │    │   │C4│C5│C6│C7│ │
│   └──┴──┴──┴──┘ │    │   └──┴──┴──┴──┘ │
└──────────────────┘    └──────────────────┘
          ↓                       ↓
        [DRAM 0]               [DRAM 1]
```

### NUMA (Non-Uniform Memory Access)

マルチソケット環境では、**core ごとに「近いメモリ」と「遠いメモリ」がある**。
- Socket 0 の core が Socket 1 の DRAM を読むと遅い
- OS は **NUMA を意識してスケジュールする**（プロセスを近いメモリの core で実行）

### キャッシュコヒーレンシ

core ごとに L1/L2 キャッシュがある → 同じメモリ位置を複数 core がキャッシュすると **不整合**が起きる。
→ ハードウェアが MESI プロトコル等で整合性を保つ：

- どちらかが書き込んだら、他の core のキャッシュを **無効化**
- 通信オーバーヘッドあり

#### False Sharing

同じキャッシュライン（通常 64 バイト）上の異なる変数を別 core から頻繁に書き込むと、**全く独立した処理なのに無効化が連発**して遅くなる：

```
[キャッシュライン 64 bytes]
[ a (4) | b (4) | c (4) | d (4) | ...                            ]
   ↑                ↑
 core 0 が a を書く     core 1 が c を書く
 → 同じキャッシュラインなので無効化合戦 → 性能急落
```

並行プログラミングでハマる典型パターン。

### SMT (Hyper-Threading)

1 つの core に **論理 core を 2 つ見せる**仕組み（Intel HT、AMD SMT）。

- 物理 4 core + HT = **OS からは 8 vCPU に見える**
- 実態は 2 論理 core が 1 物理 core を共有 → 並列度は理論値の 1.0〜1.3 倍
- メモリ待ちの隙間を別命令で埋める発想

→ **vCPU 数を 2 倍にしても性能は 2 倍にならない**ことが多い。

## 6. ユーザーモード vs カーネルモード

CPU には **特権レベル**がある（Intel x86 では Ring 0〜3）。

```
[Ring 0: カーネルモード]
  - OS カーネル（ハードウェアアクセス可）
  - デバイスドライバ
  ↓ システムコール経由
[Ring 3: ユーザーモード]
  - 普通のアプリケーション
  - ハードウェアに直接アクセス不可
```

### システムコール

ユーザープロセスがハードウェア操作（ファイル読み書き、ネットワーク等）したいとき、**システムコール**を通じてカーネルに依頼する：

```
ユーザープロセス
   │
   │ syscall (read, write, mmap, ...)
   ▼
[カーネルモードに切替]
   │
   │ ハードウェア操作
   ▼
[ユーザーモードに復帰]
   │
   ▼
ユーザープロセス（結果を受け取る）
```

- モード切替には **数百サイクル**のコスト
- 頻繁な syscall は CPU の sys 時間を増やす

### 割り込み

ハードウェア（タイマー、NIC、ディスク等）からの非同期イベントで、CPU は実行中の処理を中断して **割り込みハンドラ**を実行する。

- タイマー割り込み（通常 1000 Hz）でスケジューラが動く
- NIC のパケット受信割り込みが大量だと CPU の sys 時間が増える（"softirq" として現れる）

## 7. OS の CPU 管理（スケジューラ）

複数のプロセス・スレッドが CPU を「奪い合う」のを調停するのが **スケジューラ**。

### プロセスとスレッド

```
プロセス: アドレス空間 + リソース (FD, etc.) を持つ実行単位
   │
   ├─ スレッド 1（CPU 実行単位）
   ├─ スレッド 2
   └─ スレッド N
```

- 1 プロセスに複数スレッド
- スレッドが OS から見たスケジュール単位
- Java の場合: JVM スレッド ≒ OS スレッド（最近の JDK）

### スケジューラの基本動作

- **時分割**: 各スレッドに「タイムスライス」（例 10 ms）を割り当て
- タイムスライスが切れる or 自発的に手放す（I/O 待ち等）と、別スレッドに CPU を渡す
- このたびに **context switch**（次章）が発生

### スケジューラの方針（Linux CFS の場合）

- Completely Fair Scheduler
- 各スレッドが「公平に」CPU を得られるように仮想実行時間を追跡
- nice 値（優先度）で重みづけ可能
- リアルタイム優先度のスケジューラ（SCHED_FIFO 等）もある

## 8. Context Switch

### 何が起きるか

スケジューラが「次は別のスレッドを実行する」と決めたとき、CPU の状態（レジスタ、スタックポインタ等）を保存・復元する：

```
時刻 t0: スレッド A を実行中
   ├─ レジスタに A の状態
   ├─ L1/L2 キャッシュに A のデータ
   │
   │ タイマー割り込み発生
   ▼
時刻 t1: コンテキストスイッチ
   ├─ A のレジスタを保存
   ├─ スレッド B のレジスタを復元
   │
   ▼
時刻 t2: スレッド B を実行
   ├─ レジスタは B の状態
   ├─ L1/L2 キャッシュには A のデータ残り → B にとっては「冷たいキャッシュ」
   │   → 最初のメモリアクセスがキャッシュミスで遅い
```

### コスト

- 直接コスト: 数 μs（レジスタ保存・復元、TLB 切替）
- **間接コスト: キャッシュが冷える** → 元に戻るまで数十〜数百 μs ロス

### 多すぎる context switch のサイン

`vmstat 1` の `cs` 列：

```
$ vmstat 1
procs ---memory--- ...  --system-- ---cpu---
 r  b   swpd  free  ...   in   cs us sy id wa
 4  0      0  ...    ... 1200 3500 30 70  0  0
```

- 1 秒あたり 3500 回 context switch
- これが数万〜十数万になると **「context switch 過多」** が疑われる
- 原因: 接続数が CPU コア数を大幅に超える、頻繁な I/O 待ち、ロック待ち

## 9. CPU が「遅くなる」要因

| 要因 | 内容 | 観測指標 |
|---|---|---|
| **CPU バウンド** | 純粋な計算が多すぎる | user 時間が高い |
| **メモリ帯域** | DRAM アクセスが多すぎる | キャッシュミス率、メモリ帯域使用率 |
| **キャッシュミス** | データ局所性が悪い | L1/L2/L3 ミス率 |
| **分岐予測ミス** | データ依存の if が多い | misprediction rate |
| **Context Switch 過多** | スレッド数 >> コア数 | vmstat の cs 列 |
| **ロック競合** | 同じ mutex/semaphore を多数取り合う | spin time, lock wait |
| **I/O 待ち（CPU 観点）** | ディスク/ネット待ちで CPU 遊ぶ | top の wa (iowait) |
| **割り込み過多** | NIC/タイマー割り込み多発 | top の si (softirq) |
| **仮想化オーバーヘッド** | hypervisor 層のロス | xen 統計、steal time |
| **Steal Time（クラウド）** | 隣のテナントが CPU を奪う | top の st |
| **Thermal Throttling** | 熱で周波数が下がる | dmesg、CPU 周波数モニタ |

### 「CPU 100%」だけでは何もわからない

「DB CPU 100%」と言われても、内訳がわからないと対処できない：

| 内訳 | 真の問題 |
|---|---|
| user 100% | アプリ/DB エンジンが計算しまくり |
| sys 80% | カーネル処理が重い（syscall 過多、割り込み）|
| iowait 50% | 実は IO 待ち、CPU は遊んでる |
| steal 30% | 仮想化ホストの隣テナントに奪われてる |

→ **`top` 等で内訳を必ず確認**。

## 10. CPU 時間と並列処理能力

性能を見積もるうえで欠かせない、**「CPU 時間」「経過時間」「コア数」の単位の関係**を整理する。

### CPU 時間と経過時間の違い

| 用語 | 意味 |
|---|---|
| **CPU 時間** (CPU time) | CPU が実際にそのプロセスの処理に使った時間 |
| **経過時間** (wall-clock / elapsed time) | 開始から終了までの時計上の時間 |

I/O 待ち・ロック待ち・スケジュール待ちがあるとき、**経過時間 > CPU 時間** になる。

```
プロセスが 1 秒 走った（経過時間 = 1s）
   内訳:
     CPU で計算: 200 ms
     I/O 待ち : 600 ms
     idle    : 200 ms
   → CPU 時間 = 200 ms（経過時間の 20%）
```

### 1 コアの処理能力

ここがすべての出発点：

> **1 コアは 1 秒間に 最大 1000 ms 分の CPU 時間を提供できる**

理由は単純で「**1 秒には 1 秒分の処理しか入らない**」から。
1 コアにとって 1 秒は 1000 ms であり、それ以上の処理は時間軸に乗せられない。

### コア数と並列処理能力

複数コアあれば、並列で別々の処理を進められる：

| vCPU 数 | 1 秒あたりの提供可能 CPU 時間 |
|---:|---:|
| 1 | 1,000 ms |
| 2 | 2,000 ms |
| 4 | 4,000 ms |
| 8 | 8,000 ms |
| 16 | 16,000 ms |

→ 4 vCPU マシンなら、1 秒間に 4,000 ms 分の処理を **理論上**こなせる。

### CPU 使用率の式

```
CPU 使用率 (%) = 使用した CPU 時間 / (vCPU 数 × 経過時間) × 100
```

**例 1**: 4 vCPU マシンで 1 秒間に 2,000 ms の CPU 時間を使った
```
使用率 = 2,000 / (4 × 1,000) × 100 = 50%
```

**例 2**: 1 vCPU マシンで 100ms のクエリを毎秒 5 回実行
```
使用 CPU 時間 / 秒 = 5 × 100 = 500 ms
使用率 = 500 / (1 × 1000) × 100 = 50%
```

### スループット見積もり

逆向きに使うと「**1 件あたり N ms の処理を、CPU 何個で何 QPS 捌けるか**」が計算できる：

```
最大 QPS ≒ (vCPU 数 × 1000 ms) / 1 件あたりの処理時間 ms
```

**例**: 1 件 5ms のクエリを 4 vCPU で
```
最大 QPS ≒ 4 × 1000 / 5 = 800 QPS
```

ただし実際は **使用率 70% を超えると待ち行列効果で急速にレイテンシが伸びる**ので、運用上限はこの 50-70% に置く。

### ポイント

- 「**CPU 使用率 80%**」と「**vCPU 数 × 0.8 倍の処理時間**」は同じことを言ってる
- vCPU を倍に増やすと処理能力も**理論上**倍になる（実際は調整オーバーヘッドで線形より少し下）
- 1 処理が 1 秒以上かかるとき、その間そのコアは占有される → 他処理が並べない
- **長時間ロックを取るクエリは、他クエリが並列で来ても処理できない**ことに注意

## 11. CPU 利用率の読み方

### `top` の表示例

```
%Cpu(s):  35.2 us,  8.1 sy,  0.0 ni, 55.3 id,  0.5 wa,  0.0 hi,  0.9 si,  0.0 st
```

| 略号 | 意味 |
|---|---|
| **us** (user) | ユーザーモードでの CPU 時間。アプリ自身の計算 |
| **sy** (system) | カーネルモードでの CPU 時間。syscall 処理など |
| ni (nice) | nice 値で優先度下げたプロセス |
| **id** (idle) | 何もしてない |
| **wa** (iowait) | I/O 完了待ちで CPU は idle だが他のプロセスも実行できない状態 |
| hi (hardware interrupt) | 割り込みハンドラ |
| si (softirq) | ソフト割り込み（NIC 等） |
| **st** (steal) | クラウド・VM 環境で、ハイパーバイザに奪われた時間 |

### Load Average

```
load average: 8.34, 6.12, 4.05
```

- 過去 1 分・5 分・15 分の **実行可能 + 実行中スレッド数**の平均
- vCPU 数を超えてる = 待ち行列ができてる
- 例: 4 vCPU マシンで load 8 → 半分のスレッドが待たされている

### CPU 使用率の経時変化

- 平均 50% でも、**ピークで 100% に当たれば**ユーザー体感は悪化
- → グラフで「ピーク張り付き時間」を見る

## 13. アプリケーション開発者が意識すること

### データ局所性

「メモリ上で近くにあるデータを連続アクセス」する設計が CPU に優しい：

```python
# 局所性が良い（同じ配列を順に処理）
for i in range(N):
    data[i] = compute(data[i])

# 局所性が悪い（散らばったオブジェクトを跨ぐ）
for obj in objects:
    obj.field = compute(other_dict[obj.id].something)
```

### 並列化と Amdahl の法則

「**直列処理の部分があると、並列度を上げても頭打ち**」という法則：

```
speedup = 1 / ((1 - P) + P / N)

P = 並列化可能な割合
N = プロセッサ数
```

例: 90% 並列化できる処理を 16 core で動かす → 最大 speedup ≈ 6.4 倍（16 倍にはならない）

### ロックの粒度

- 大きなロック → 競合多発、context switch 増
- 細かすぎるロック → ロック取得コスト累積
- ロックフリーアルゴリズム（CAS 等）も選択肢

### False Sharing 回避

- 並行アクセスされる変数を **キャッシュライン跨ぎ**にする（padding）
- Java の `@Contended` アノテーション

### syscall を減らす

- 細切れの read/write より、まとまった量を一度に
- ファイルディスクリプタの使い回し（接続プール等）
- バッファリング

### JIT 最適化を活かす

- 同じコードパスを繰り返し通すと JIT が最適化する
- 起動直後は遅く、温まると速くなる（warm-up）

## 14. クラウド環境での CPU

### vCPU の意味

- vCPU = SMT (Hyper-Threading) の **論理 core 1 つ**
- 物理 core の半分の演算能力と考えると安全
- 「2 vCPU」と言われたら物理的には 1 core 相当

### CPU クレジット（Burstable インスタンス）

AWS t3 / t4g 系などの Burstable インスタンスは **CPU クレジット制**：

- 通常時はベースラインの CPU 使用率（例 t3.medium で 20%）
- それ以上使うとクレジット消費
- クレジット枯渇すると **ベースラインまで強制スロットル**

→ **本番 DB に Burstable は危険**。安定稼働には r 系（メモリ最適化）or m 系（汎用）推奨。

### Steal Time

仮想化ホスト上で、**同じ物理サーバの他テナントが CPU を使っている時間**：

```
%Cpu(s):  ...  st 15.0
```

- `st` が常時 5% 以上 → noisy neighbor 問題
- インスタンスタイプ変更で改善できることが多い

### 専用インスタンス / 専用ホスト

- Dedicated Instance: 他テナントと物理サーバを共有しない
- Dedicated Host: 物理サーバを丸ごと占有
- → Steal Time をゼロに近づけられる、コンプライアンス用途も

## 15. まとめ・関連ドキュメント

### キーメッセージ

1. **CPU は単純な計算機ではない**。キャッシュ階層、パイプライン、分岐予測、OoO 実行など多くの仕組みで「実効性能」が決まる
2. **「CPU 100% 張り付き」だけでは何もわからない**。内訳（us/sy/wa/st）を見ないと対処できない
3. **メモリ局所性とキャッシュが性能を支配**する（DRAM は L1 の 100 倍遅い）
4. **接続数 >> コア数 = context switch 地獄**。アプリの接続上限は慎重に
5. **クラウドでは Steal Time と CPU クレジット**を意識する
6. **計測してから対処**。`top` / `vmstat` / `perf` で内訳を見る
