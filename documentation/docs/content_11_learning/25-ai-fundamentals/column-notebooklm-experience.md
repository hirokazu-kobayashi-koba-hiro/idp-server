# コラム: Google NotebookLMを使ってみた - AIで変わる学習と調査

---

## はじめに

「大量の資料を読まなきゃいけないけど、時間がない」「複雑な技術文書を理解したい」。そんな悩みを持つ方に朗報です。GoogleのNotebookLMは、あなたの資料を理解するパーソナルAIアシスタントです。本コラムでは、実際の使用体験を通じて、NotebookLMの可能性と限界を探ります。

## 実際に使ってみた: ユースケース別レビュー

### ユースケース1: 技術仕様書の理解

**シナリオ**: OAuth 2.0のRFC（100ページ超）を読みたいが時間がない

**手順**:
```
1. NotebookLMにアクセス
   └─ https://notebooklm.google.com/

2. 新しいノートブック作成
   └─ 「OAuth 2.0学習ノート」

3. ソース追加
   ├─ RFC 6749（OAuth 2.0）のPDF
   ├─ RFC 6750（Bearer Token）のPDF
   └─ セキュリティのベストプラクティス文書

4. 質問開始
```

---

**実際の対話例**:

```
私: 「OAuth 2.0の認可コードフローを初心者向けに説明して」

NotebookLM:
「OAuth 2.0の認可コードフローは、最も安全な認可方式です。

 流れ:
 1. ユーザーが認可サーバーにリダイレクト
 2. ユーザーが許可を承認
 3. 認可コードが発行される
 4. アプリケーションがコードをトークンと交換

 この方式の利点は、アクセストークンがブラウザに露出しない点です。

 出典: RFC 6749, Section 4.1」
     ↑
  ページ番号付きで引用元を明示
```

**所感**:
- ✅ **正確性**: RFCの記載に忠実
- ✅ **速度**: 100ページを数秒で検索
- ✅ **引用**: どのページに書いてあるか明確
- ⚠️ **深さ**: 文書にない情報は提供されない（これは長所でもある）

### ユースケース2: 会議資料の事前予習

**シナリオ**: 明日の経営会議、50ページの資料を今から読む時間がない

**手順**:
```
1. NotebookLM起動

2. ソース追加
   ├─ 経営会議資料（PDF）
   ├─ 前回議事録
   └─ 関連する財務レポート

3. AI要約依頼
   「この資料の重要ポイントを3つ教えて」

4. 想定質問の準備
   「Q&A形式で想定される質問と回答を5つ作成して」
```

---

**NotebookLMの出力**:

```
■ 重要ポイントTOP3

1. Q4売上が目標未達（-8%）
   - 理由: 競合の新製品投入
   - 対策: 新価格戦略を提案（資料p.12）

2. AI事業への投資提案（¥500M）
   - 期待ROI: 3年で2.5倍
   - リスク: 人材確保の困難さ（資料p.28）

3. 組織再編の提案
   - 開発部門を2分割
   - 目的: 意思決定スピード向上（資料p.45）

■ 想定Q&A

Q1: なぜQ4が未達だったのか？
A: 競合X社が9月に新製品を投入し、当社の主力
   セグメントで価格競争が激化したため。
   （出典: 資料p.12-14）

Q2: AI事業の成功確率は？
A: 資料では68%と試算。ただし最初の1年で顧客
   100社獲得が前提条件。（出典: 資料p.30）
```

**所感**:
- ✅ **時短**: 50ページを5分で把握
- ✅ **的確**: 本当に重要な部分を抽出
- ✅ **準備万端**: 会議で質問されても即答できる
- 💡 **新発見**: 自分が見落としていた関連性を指摘してくれる

## NotebookLMの強み

### 1. ソース引用の正確性

```
他のLLMとの比較:

ChatGPT:
質問: 「OAuth 2.0の認可フローは？」
回答: 「認可フローには4種類あります...」
出典: ❌ 不明（ハルシネーションの可能性）

NotebookLM:
質問: 「OAuth 2.0の認可フローは？」
回答: 「4種類のフローがあります:
       1. 認可コードフロー（RFC 6749, p.24）
       2. インプリシットフロー（p.31）
       ...」
出典: ✅ RFC 6749の具体的なページを明示

→ 信頼性が段違い
```

---

### 2. 複数文書の横断検索

```
ユースケース: 自社の過去3年分の議事録を検索

質問: 「AI投資について過去にどんな議論があった？」

NotebookLM:
「AI投資に関する議論が4回ありました:

 1. 2022年3月: 見送り
    理由: ROI不明確（2022-Q1議事録p.5）

 2. 2023年5月: 再検討
    理由: 競合が導入開始（2023-Q2議事録p.8）

 3. 2023年11月: パイロット決定
    予算: ¥50M（2023-Q4議事録p.12）

 4. 2024年8月: 本格投資
    予算: ¥500M（2024-Q3議事録p.15）」

```

### 3. 非公開・最新情報への対応

```
一般的なLLMの限界:

├─ 非公開情報: アクセス不可
└─ 社内文書: 読めない

NotebookLM:
├─ 最新: 今日の資料でもOK
├─ 非公開: 自分だけの資料を使用
└─ 社内文書: 何でもアップロード可能

例:
「2026年1月23日の取締役会資料」もOK
「社外秘のM&A検討資料」もOK（※セキュリティ考慮は必要）
```

## NotebookLMの限界と注意点

### 1. 情報の範囲

```
✅ できること:
「アップロードした資料に書いてあることを探す・要約する」

❌ できないこと:
「資料にない情報を推測する」

例:
質問: 「この技術の将来性は？」
資料: 現状の説明のみ

NotebookLM:
「資料には将来性に関する記載がありません」

→ 正直だが、洞察は得られない
```

---

### 2. クリティカルシンキングは人間の仕事

```
NotebookLMは資料を要約するが、批判的に分析しない

例: 企業のIR資料
├─ NotebookLM: 「売上+20%と記載されています」
└─ 人間の分析: 「でも利益率は悪化。本当に健全？」

→ 数字の裏を読むのは人間の役割
```

---

### 3. セキュリティとプライバシー

**重要な注意点**:
```
⚠️  アップロードしたデータは Google に送信される

対策:
❌ 機密情報をそのままアップロードしない
  - 顧客個人情報
  - 社外秘の財務データ
  - 未公開のM&A情報

✅ 匿名化・マスキングしてから使用
  - 固有名詞を伏せ字に
  - 具体的な数値を範囲に置換
  - または社内専用RAGシステムを構築
```

**Googleのポリシー（2025年1月時点）**:
- アップロードデータは**モデル学習には使用しない**（明言）
- ただしGoogleのサーバーに保存される
- 企業利用は要検討

---

### 4. 音声の多言語対応 🎉

```
✅ 2025年4月29日アップデート: 日本語対応開始！

現状（2026年1月）:
├─ Audio Overview: 50以上の言語対応
├─ 日本語資料 → 日本語音声で解説
├─ 自然な日本語会話で聞きやすい
└─ 言語設定: Googleアカウントまたは出力言語設定

対応言語（主要）:
日本語、英語、中国語、韓国語、スペイン語、
フランス語、ドイツ語、イタリア語など

注意点:
- 多言語版は英語版よりやや短め
- 今後さらなる改善予定
```

## 実践的な使い方のコツ

### 1. 良い質問の仕方

**❌ 曖昧な質問**:
```
「この資料について教えて」
→ 範囲が広すぎて役立たない
```

**✅ 具体的な質問**:
```
「第3章に記載されているリスクを3つ挙げて、
 それぞれの対策も教えて」
→ 明確な回答が得られる
```

---

### 2. 段階的な深掘り

```
質問の流れ:

Step 1（全体把握）:
「この資料の主要なポイントを3つ教えて」

Step 2（興味のある部分に深掘り）:
「ポイント2の『AI投資』について、
 具体的な投資先と金額を教えて」

Step 3（関連情報）:
「過去の投資事例と比較すると、
 今回の投資は大きい？小さい？」

Step 4（アクション）:
「この投資を成功させるために
 必要な条件は何？」
```

---

### 3. ソースの使い分け

```
NotebookLMに追加すべき資料:
✅ 信頼できる一次情報
  - 公式文書、論文、契約書等

⚠️  注意が必要:
  - ブログ記事（質が様々）
  - SNS投稿（信頼性低い）

組み合わせ例:
├─ 公式ドキュメント（基礎理解）
├─ 論文（深い理解）
└─ 実装例（ベストプラクティス）

→ 多角的に理解できる
```

## 実践例: idp-server開発での活用

### シナリオ: OAuth 2.0実装の調査

**手順**:
```
1. NotebookLMに関連資料アップロード
   ├─ RFC 6749（OAuth 2.0 Core）
   ├─ RFC 6750（Bearer Token）
   ├─ RFC 7636（PKCE）
   ├─ idp-server既存設計文書
   └─ 競合製品のドキュメント

2. 質問フロー
   ステップ1: 「OAuth 2.0の必須要件は？」
   ステップ2: 「idp-serverで未実装の機能は？」
   ステップ3: 「競合製品と比較した当社の強み・弱みは？」
   ステップ4: 「PKCE実装の注意点は？」

3. 実装計画作成
   NotebookLMの回答を基に、実装優先順位を決定
```

**成果**:
```
調査時間: 2日 → 4時間（75%削減）

発見:
- RFC必須要件のうち3つが未実装と判明
- 競合は全社PKCE対応済み → 優先度高
- idp-serverの強み（マルチテナント）を再確認

→ データドリブンな実装計画が立てられた
```
