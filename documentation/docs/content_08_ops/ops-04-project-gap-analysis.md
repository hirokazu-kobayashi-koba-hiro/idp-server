# プロジェクト広域ギャップ分析

> 分析日: 2026-03-23

プロジェクト全体を7つの観点で整理し、現状の強みとギャップを明らかにする。

---

## A. 可観測性（Observability） — 最大のギャップ

| 項目 | 現状 | ギャップ |
|------|------|---------|
| ログ | ✅ JSON構造化 + MDC + モジュール別制御 | — |
| ヘルスチェック | ✅ Actuator + テナント別 | — |
| **メトリクス** | ❌ Prometheus/Micrometer なし | 認証成功率、トークン発行数、レイテンシ分布などが計測できない |
| **分散トレーシング** | ❌ OpenTelemetry/Jaeger なし | テナント横断のリクエストフロー追跡ができない |
| **アラート** | ❌ 定義なし | 異常検知が外部依存 |

`TenantStatistics` クラスは存在するが、メトリクスAPIとして公開されていない。ログとヘルスチェックだけでは本番運用で限界がある。

---

## B. レジリエンス（Resilience）

| 項目 | 現状 | ギャップ |
|------|------|---------|
| リトライ | ✅ HttpRetryStrategy（指数バックオフ） | — |
| Graceful Shutdown | ✅ K8s対応 | — |
| SSRF防御 | ✅ SsrfProtectedHttpClient | — |
| **サーキットブレーカー** | ❌ なし | 外部サービス障害時のカスケード障害リスク |
| レートリミット | API Gateway側の責務 | テナント単位の流量制御はNginx/Kong/ALB等で対応。アプリ層にはBruteForceProtection（パスワード認証のロック）が存在 |

リトライはあるがサーキットブレーカーがないので、外部eKYCサービスやSMS送信サービスが長時間障害を起こすと、スレッドプールが枯渇するリスクがある。

### サーキットブレーカーとは

外部サービスが障害を起こしたときに、こちらまで巻き込まれないようにする仕組み。

```
正常時                障害検知              一定時間後
┌────────┐  失敗が閾値超え  ┌────────┐  少しだけ通す  ┌────────────┐
│ CLOSED │ ───────────→ │  OPEN  │ ──────────→ │ HALF-OPEN  │
│ 全部通す │              │ 全部遮断 │              │ 様子見     │
└────────┘              └────────┘              └────────────┘
     ↑                                            │    │
     └────── 成功 → CLOSED に戻る ─────────────────┘    │
                                                       │
              失敗 → OPEN に戻る ──────────────────────┘
```

対象になりそうな箇所:

| 外部連携 | 現状 | リスク |
|---------|------|-------|
| eKYCサービス (http_request) | リトライのみ | タイムアウト×リトライ回数分スレッド占有 |
| SMS送信 (http_request) | リトライのみ | 同上 |
| Email送信 (http_request) | リトライのみ | 同上 |
| Webhookフック送信 | リトライのみ | フック先障害で全イベント処理停滞 |
| Federation (外部IdP) | リトライのみ | 外部IdP障害でログイン全停止 |

## F. ドキュメントの知識分散

| ドキュメント種別 | 場所 | 問題 |
|----------------|------|------|
| スキル（40+） | `.claude/skills/` | AI向け。人間が直接読むには最適化されていない |
| EXPERIMENTS (8本) | `config/templates/` | 設計判断の宝庫だがスキル・開発者ガイドに反映されていない（user_code問題） |
| VERIFY-CONFIG-CHANGES (4本) | `config/templates/` | 同上 |
| 開発者ガイド | `documentation/` | EXPERIMENTSの知見が未反映 |

**「知識はあるが、適切な場所にない」** 問題。EXPERIMENTSに書かれた設計判断（user_code = password 等）が開発者ガイドやスキルに伝搬していない。

```
            影響度 高

   メトリクス ●          ● サーキットブレーカー
  (Micrometer)           (Resilience4j)
              │
              │          ● レートリミット
              │
 ─────────────┼─────────────── 導入コスト
              │         高
  CHANGELOG ● │
              │    ● クライアントSDK

              │    ● 分散トレーシング
              │      (OpenTelemetry)
              │
            影響度 低
```
