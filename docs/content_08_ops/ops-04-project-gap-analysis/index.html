プロジェクト広域ギャップ分析

分析日: 2026-03-23

プロジェクト全体を7つの観点で整理し、現状の強みとギャップを明らかにする。

A. 可観測性（Observability） — 最大のギャップ​

項目現状ギャップ
ログ✅ JSON構造化 + MDC + モジュール別制御—
ヘルスチェック✅ Actuator + テナント別—
メトリクス❌ Prometheus/Micrometer なし認証成功率、トークン発行数、レイテンシ分布などが計測できない
分散トレーシング❌ OpenTelemetry/Jaeger なしテナント横断のリクエストフロー追跡ができない
アラート❌ 定義なし異常検知が外部依存

TenantStatistics クラスは存在するが、メトリクスAPIとして公開されていない。ログとヘルスチェックだけでは本番運用で限界がある。

B. レジリエンス（Resilience）​

項目現状ギャップ
リトライ✅ HttpRetryStrategy（指数バックオフ）—
Graceful Shutdown✅ K8s対応—
SSRF防御✅ SsrfProtectedHttpClient—
サーキットブレーカー❌ なし外部サービス障害時のカスケード障害リスク
レートリミットAPI Gateway側の責務テナント単位の流量制御はNginx/Kong/ALB等で対応。アプリ層にはBruteForceProtection（パスワード認証のロック）が存在

リトライはあるがサーキットブレーカーがないので、外部eKYCサービスやSMS送信サービスが長時間障害を起こすと、スレッドプールが枯渇するリスクがある。

サーキットブレーカーとは​

外部サービスが障害を起こしたときに、こちらまで巻き込まれないようにする仕組み。

正常時                障害検知              一定時間後
┌────────┐  失敗が閾値超え  ┌────────┐  少しだけ通す  ┌────────────┐
│ CLOSED │ ───────────→ │  OPEN  │ ──────────→ │ HALF-OPEN  │
│ 全部通す │              │ 全部遮断 │              │ 様子見     │
└────────┘              └────────┘              └────────────┘
     ↑                                            │    │
     └────── 成功 → CLOSED に戻る ─────────────────┘    │
                                                       │
              失敗 → OPEN に戻る ──────────────────────┘

対象になりそうな箇所:

外部連携現状リスク
eKYCサービス (http_request)リトライのみタイムアウト×リトライ回数分スレッド占有
SMS送信 (http_request)リトライのみ同上
Email送信 (http_request)リトライのみ同上
Webhookフック送信リトライのみフック先障害で全イベント処理停滞
Federation (外部IdP)リトライのみ外部IdP障害でログイン全停止

C. テスト・品質保証​

項目現状ギャップ
E2Eテスト✅ Jest 4層構造（176スイート）CIBA基本フロー未移植
OWASP依存関係チェック✅ CVSSスコア9以上でビルド失敗—
Spotless✅ Google Java Format—
コードカバレッジ✅ JaCoCo 導入済み (2026-03-23)全体LINE 7.5%（platform層40-92%、usecases/adapters層は0% = E2Eでカバー）
SAST✅ CodeQL 導入済み (2026-03-23)—
Pre-commitフック❌ なしCI前にフォーマット・lint違反を防げない
性能回帰テスト❌ CI未統合k6は手動実行のみ

D. 開発者エコシステム​

項目現状ギャップ
OpenAPI仕様✅ 17ファイル—
運用ドキュメント✅ 7本の商用デプロイガイド—
スキル体系✅ 40+スキル—
CONTRIBUTING.md✅ あり—
クライアントSDK❌ なし利用者がcurl/自前実装するしかない
CHANGELOG不要GitHub Releases のリリースノート（PR一覧 + Full Changelog リンク自動生成）で代替済み
API バージョニング不要OAuth/OIDC仕様準拠のため破壊的変更が起きにくい。必要時に /v2/ を切るだけで十分

E. シークレット管理・本番運用​

項目現状ギャップ
環境変数テンプレート✅ .env.example（67変数）—
ファイルベースシークレット✅ SECRETS_DIR + migrate-secrets.sh—
K8sマニフェスト✅ ローカル開発用—
Vault/AWS Secrets Manager❌ 未統合K8sマニフェストに「本番では External Secrets Operator を使え」と注記あり
フィーチャーフラグ不要テナント設定 + モジュール分離で代替済み。少人数チームでコントロール可能

F. ドキュメントの知識分散​

ドキュメント種別場所問題
スキル（40+）.claude/skills/AI向け。人間が直接読むには最適化されていない
EXPERIMENTS (8本)config/templates/設計判断の宝庫だがスキル・開発者ガイドに反映されていない（user_code問題）
VERIFY-CONFIG-CHANGES (4本)config/templates/同上
開発者ガイドdocumentation/EXPERIMENTSの知見が未反映

「知識はあるが、適切な場所にない」 問題。EXPERIMENTSに書かれた設計判断（user_code = password 等）が開発者ガイドやスキルに伝搬していない。

G. 優先度マトリクス​

影響度と導入コストで整理:

            影響度 高
              │
   メトリクス ●          ● サーキットブレーカー
  (Micrometer)           (Resilience4j)
              │
              │          ● レートリミット
              │
 ─────────────┼─────────────── 導入コスト
              │         高
  CHANGELOG ● │
              │    ● クライアントSDK
 Pre-commit ● │
              │    ● 分散トレーシング
              │      (OpenTelemetry)
              │
            影響度 低

すぐ始められて効果が高いもの​

  1. Micrometer + Prometheus — Spring Boot Starter追加 + @Timed で主要エンドポイントの計測開始
  2. JaCoCo — ✅ 導入済み (2026-03-23)
  3. CHANGELOG — GitHub Releases で代替済み

戦略的に取り組むべきもの​

  1. サーキットブレーカー — 外部サービス連携（eKYC, SMS, Webhook）の障害耐性
  2. EXPERIMENTSの知識をスキル・開発者ガイドに反映 — レビュー品質向上
